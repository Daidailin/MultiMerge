# 时间处理系统架构概述

## 系统整体架构

MultiMerge 项目的时间处理系统由两个核心类组成：TimePoint 和 TimeParser，它们共同提供了完整的时间表示、解析和运算功能。

```
┌─────────────────────────────────────────────────────────────┐
│                     时间处理系统                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────┐          ┌─────────────────────┐  │
│  │    TimePoint        │          │    TimeParser       │  │
│  │  (时间表示与运算)    │◄─────────│  (字符串解析)        │  │
│  │                     │  返回     │                     │  │
│  │  - 时/分/秒/小数     │  TimePoint│  - 单例模式         │  │
│  │  - 毫秒/微秒/纳秒    │          │  - 多格式支持       │  │
│  │  - 标准化处理        │          │  - 可扩展           │  │
│  │  - 单位转换          │          │                     │  │
│  │  - 比较/算术运算     │          │                     │  │
│  └─────────────────────┘          └─────────────────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 设计哲学

### 1. 职责分离原则
- **TimePoint**: 专注于时间的内部表示和运算
- **TimeParser**: 专注于字符串到时间对象的转换

### 2. 统一微秒基准

所有时间运算都转换为纳秒进行，确保精度一致性和计算准确性。

### 3. 自动标准化
时间对象在创建和运算后自动标准化到合法范围，避免非法状态。

### 4. 结果模式错误处理
使用 ParseResult 封装成功/失败结果，避免异常的使用，提高性能和可预测性。

## 核心数据流

### 解析流程
```
输入字符串 → TimeParser::parse() → 格式匹配 → 解析函数 → TimePoint 对象
```

### 运算流程
```
TimePoint → toNanoseconds() → 纳秒值 → 运算 → fromNanoseconds() → 标准化 TimePoint
```

### 输出流程
```
TimePoint → toString() → 格式化字符串（自动补零）
```

## 清理版架构（当前版本）

### 支持的格式
当前版本（2.0 清理版）只支持两种格式：
1. **HH:MM:SS:SSS** - 冒号分隔毫秒格式（3 位小数）
2. **HH:MM:SS:SSSSSS** - 冒号分隔微秒格式（6 位小数）

### 精度自动推断
- 1-3 位小数 → 毫秒精度
- 4-6 位小数 → 微秒精度
- **不支持纳秒精度**

### 输入输出规范
- **输入**: 支持补零和不补零（如 "01:02:03:5" 和 "1:2:3:5"）
- **输出**: 统一左补零（如 "1:2:3:5" → "01:02:03:005"）

### 设计决策
**为什么移除纳秒支持？**
- 实际应用场景中毫秒和微秒精度已足够
- 简化代码逻辑，降低维护成本
- 减少精度转换带来的复杂性
- 纳秒仍作为内部计算单位使用

## 关键设计决策

### 1. 为什么使用纳秒作为内部基准？
- 纳秒是最高精度，可以无损转换到其他精度
- 统一使用纳秒避免精度混乱
- int64_t 可以容纳一天的纳秒数（86.4 万亿）

### 2. 为什么使用 24 小时循环而非抛出异常？
- 时间序列数据经常跨天，循环处理更符合实际需求
- 使用取模公式优雅处理溢出，避免复杂的条件判断
- 提高性能，减少异常处理开销

### 3. 为什么解析时不补零，输出时才补零？
- 保持原始数值不变，避免计算误差
- 输出格式化是展示层职责，不应影响内部表示
- 简化解析逻辑，提高性能

### 4. 为什么使用单例模式？
- 解析器是无状态的，全局共享安全
- 避免重复注册格式的开销
- 提供统一的格式管理入口

## 扩展机制

### 自定义格式注册
```cpp
TimeFormatDescriptor customFormat(
    "CustomFormat",
    R"(^\d{1,2}-\d{1,2}-\d{1,2}$)",
    [](const QString& str) -> ParseResult {
        // 自定义解析逻辑
        return ParseResult(timePoint);
    }
);
TimeParser::instance().registerFormat(customFormat);
```

### 错误处理扩展
```cpp
enum class ParseErrorCode {
    Success = 0,
    InvalidFormat,
    InvalidHour,
    InvalidMinute,
    InvalidSecond,
    InvalidFraction,
    OutOfRange,
    EmptyInput,
    UnknownError
};
```

## 性能特性

### 时间复杂度
- 解析：O(n)，n 为注册格式数量
- 标准化：O(1)
- 单位转换：O(1)
- 比较运算：O(1)

### 空间复杂度
- TimePoint: 固定大小（3 个 int + 1 个 int64_t + 1 个 enum = 24 字节）
- ParseResult: 固定大小
- TimeParser: O(m)，m 为注册格式数量

## 线程安全性

### TimePoint
- 值类型，复制安全
- 无共享状态，天然线程安全

### TimeParser
- 单例实例线程安全（C++11 静态局部变量保证）
- 动态注册格式时非线程安全（需外部同步）
- 解析操作只读，多线程安全

## 测试策略建议

### 单元测试重点
1. **边界条件**: 00:00:00, 23:59:59, 溢出情况
2. **精度转换**: 毫秒↔微秒转换
3. **标准化**: 正负溢出、进位借位
4. **解析**: 各种格式、错误输入
5. **运算**: 加减运算、跨天情况

### 集成测试重点
1. 文件读取→解析→合并→输出的完整流程
2. 大文件性能测试
3. 异常数据容错测试

## 未来演进方向

### 可能的增强
1. **时区支持**: 添加时区转换功能
2. **日期支持**: 扩展为完整日期时间
3. **相对时间**: 支持"1 小时后"等相对时间表达
4. **本地化**: 支持不同地区的时间格式

### 性能优化方向
1. 正则表达式预编译缓存
2. 字符串解析优化（避免正则表达式）
3. 内存池管理（频繁创建 TimePoint 对象时）

## 总结

时间处理系统通过清晰的职责分离、统一的纳秒基准和自动标准化机制，提供了高效、准确的时间处理能力。**当前版本（2.0 清理版）专注于毫秒和微秒两种实用精度**，移除了纳秒支持，保持了代码的简洁性和可维护性，同时保留了扩展的可能性。纳秒仅作为内部计算单位使用，不对外暴露。
